# RB-01 — EDI 820 remittance advice   (keel-map research, 2026-05-28)

**Why it matters here.** ENG-04 · instance: EDI 820 (Integrations). Blocks the
Payment Matching module: remittance ingestion is its input. Generic question it
replaces: "confirm the EDI integration specifications" (Q-12 → Q-31).

**What the research found** *(sourced; fixture URLs illustrative)*
- EDI 820 is the X12 Payment Order/Remittance Advice transaction set; current
  bank implementations are predominantly X12 5010, with 4010 still common.
  (x12.org transaction-set reference, accessed 2026-05-28)
- Transport is bank-specific: SFTP drop, AS2, or VAN. Acknowledgment via 997
  functional ack is optional per trading-partner agreement.
  (RBC EDI onboarding guide, accessed 2026-05-28)

**The decision space**
- X12 5010 vs 4010 vs bank-proprietary CSV — affects parser scope.
- SFTP daily drop vs AS2 push — affects infrastructure and retry handling.
- 997 acks on/off — affects how missing-file detection works.

**Questions for the lead** *(each with example answers)*
1. Which EDI 820 version does the bank require — e.g. **X12 5010**, **X12 4010**, or a **bank-proprietary CSV**?
2. How do files arrive — e.g. **SFTP daily drop**, **AS2 push**, or **VAN mailbox**?
3. Are 997 functional acknowledgments expected — e.g. **yes, per file**, or **no acks**?

**Suggested default assumption** — X12 5010, single bank (RBC), SFTP daily
drop, no 997s. *Impact-if-wrong:* parser rework + trading-partner re-test,
est. 1–2 weeks (→ RAID-A07).
